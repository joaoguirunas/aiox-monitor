# Story 1.3 — Endpoint POST /api/events

## Story

**Como** hook Python rodando em `~/.claude/hooks/`,
**Quero** enviar eventos via HTTP POST para `http://localhost:8888/api/events`,
**Para que** cada ação do Claude Code seja registrada no banco SQLite local.

## Acceptance Criteria

- [x] **AC1** — `POST /api/events` aceita JSON com campos: `hook_type`, `project_path`, `project_name`, `agent_name`, `tool_name`, `input`, `output`, `timestamp`, `terminal_pid`, `terminal_session_id`
- [x] **AC2** — Endpoint faz UPSERT de `project` baseado em `project_path`
- [x] **AC3** — Endpoint faz UPSERT de `agent` baseado em `(project_id, agent_name)`
- [x] **AC4** — Endpoint faz UPSERT de `terminal` baseado em `(project_id, terminal_pid)`
- [x] **AC5** — Endpoint insere evento na tabela `events` com todos os campos mapeados corretamente
- [x] **AC6** — Evento do tipo `Stop` ou `SubagentStop` marca o terminal como `inactive` e fecha a session ativa
- [x] **AC7** — Endpoint responde `200 OK` com `{ success: true, event_id: number }` em caso de sucesso
- [x] **AC8** — Endpoint responde `400` se payload inválido (sem `project_path` ou `hook_type`)
- [x] **AC9** — Falha no processamento NÃO retorna 500 — retorna `200` com `{ success: false, error: string }` (não deve bloquear o hook)

    const payload = await request.json();
    if (!payload.project_path || !payload.hook_type) {
      return Response.json({ success: false, error: 'Missing required fields' }, { status: 400 });
    }
    try {
      const result = processEvent(payload);
      return Response.json({ success: true, event_id: result.id });
    } catch (error) {
      // Retorna 200 para não bloquear o hook Python
      return Response.json({ success: false, error: String(error) });
    }
  }

## Dev Notes

### AC9 — Por que retornar 200 em erros?
O hook Python tem timeout de 2 segundos e falha silenciosa. Se o servidor retornar 5xx, o hook pode logar stderr e potencialmente atrasar o Claude Code. Retornar 200 com `{ success: false }` é mais seguro.

### Ordem de processamento no event-processor
```
payload recebido
  → detectProject(path, name)      → upsertProject → retorna Project
  → trackTerminal(pid, sessionId)  → upsertTerminal → retorna Terminal
  → trackAgent(agentName)          → upsertAgent → retorna Agent
  → insertEvent(...)               → retorna Event
  → se Stop/SubagentStop:
      deactivateTerminal(pid)
      closeSession(activeSessionId)
```

### Mapeamento hook_type → EventType
```typescript
const TYPE_MAP: Record<string, EventType> = {
  'PreToolUse': 'PreToolUse',
  'PostToolUse': 'PostToolUse',
  'UserPromptSubmit': 'UserPromptSubmit',
  'Stop': 'Stop',
  'SubagentStop': 'SubagentStop',
};
```

### Inicialização do setInterval do cleanup
O `setInterval` em `terminal-tracker.ts` precisa ser inicializado apenas uma vez. Em Next.js App Router, isso pode ser feito num module-level side effect ou num custom server. Para MVP, module-level é suficiente.
