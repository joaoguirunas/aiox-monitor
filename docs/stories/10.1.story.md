# Story 10.1 — Terminal Data Bridge + Permission Badge + Tool Detail

## Story

**Como** utilizador do aiox-monitor,
**Quero** ver no Kanban um badge pulsante quando um agente está à espera de permissão e o detalhe da tool em execução (ex: "Reading src/lib/db.ts"),
**Para que** saiba instantaneamente quando preciso intervir e o que cada agente está a fazer em concreto, sem trocar de aba.

## Context

A Story 6.1 adicionou `waiting_permission` e `current_tool_detail` à tabela `terminals`, mas o Kanban consome apenas a tabela `agents` via `GET /api/agents` (sem `expand=terminals`). O broadcast WS `agent:update` em `agent-tracker.ts:33,44` envia apenas campos da tabela `agents` (`status`, `current_tool`). Os dados existem no backend mas nunca chegam ao frontend Kanban.

**Root cause:** O Kanban não acede a dados de terminal. Esta story implementa o bridge.

**Decisão arquitectural (PM):** Opção B (enriquecer WS) + Opção A (fallback fetch).

## Acceptance Criteria

### Task 1: Enriquecer WS `agent:update` com dados de terminal

- [x] **AC1** — `agent-tracker.ts` → Antes de cada broadcast `agent:update`, consultar o terminal mais recente do agente (`terminals` WHERE `agent_name` = agent.name AND `project_id` = projectId ORDER BY `last_active` DESC LIMIT 1) e incluir `waiting_permission` e `current_tool_detail` no payload
- [x] **AC2** — Payload do `agent:update` WS passa a incluir: `{ type: 'agent:update', agent: { ...agent, waiting_permission, current_tool_detail }, projectId }`
- [x] **AC3** — Se não houver terminal activo para o agente, enviar `waiting_permission: 0` e `current_tool_detail: null` (fallback seguro)

### Task 2: Fetch inicial com dados de terminal

- [x] **AC4** — `useAgents.ts:20` → Alterar fetch URL de `/api/agents` para `/api/agents?expand=terminals` para que o estado inicial inclua `waiting_permission` e `current_tool_detail`
- [x] **AC5** — `useAgents.ts:49` → No handler WS para agentes novos, preservar `waiting_permission` e `current_tool_detail` do payload (não usar defaults hardcoded)

### Task 3: Actualizar tipo Agent

- [x] **AC6** — `types.ts` → Adicionar campos opcionais à interface `Agent` (linhas 9-17): `waiting_permission?: 0 | 1` e `current_tool_detail?: string`

### Task 4: Badge `waiting_permission` no AgentCard

- [x] **AC7** — Variante `card`: Quando `waiting_permission === 1`, exibir badge pulsante amber/vermelho com texto "Aguarda permissão" posicionado junto ao tool detail
- [x] **AC8** — Variante `chip`: Quando `waiting_permission === 1`, exibir indicador visual compacto (dot amber pulsante) junto ao nome do agente
- [x] **AC9** — Animação do badge: usar `animate-pulse` do Tailwind com cor `amber-400` para garantir visibilidade imediata
- [x] **AC10** — Quando `waiting_permission === 0` ou `undefined`, badge ausente (sem espaço reservado)

### Task 5: Tool Detail enriquecido no AgentCard

- [x] **AC11** — Variante `card`: Substituir display de `agent.current_tool` por `agent.current_tool_detail` quando disponível
- [x] **AC12** — Fallback: Se `current_tool_detail` é `null`/`undefined`, manter exibição de `current_tool` como actualmente
- [x] **AC13** — Truncar `current_tool_detail` a 40 caracteres com ellipsis para não quebrar layout do card

### Task 6: Validação

- [x] **AC14** — Build passa sem erros (`npm run build`)
- [x] **AC15** — Render do Kanban com 30 agentes < 100ms
- [x] **AC16** — Badge aparece em < 2 segundos após terminal entrar em estado `waiting_permission = 1`
