# Story 11.5 — Persistência de Terminais da Sala de Comando no Banco de Dados

## Story

**Como** utilizador da Sala de Comando,
**Quero** que os terminais abertos sejam persistidos no banco de dados SQLite,
**Para que** ao recarregar a página ou reiniciar o servidor, os terminais e suas configurações sejam restaurados automaticamente sem perda de estado.

## Context

**Problema raiz identificado:** O estado dos terminais da Sala de Comando é guardado em dois lugares voláteis:
1. **localStorage do browser** (`LS_KEY = 'aiox-command-room-projects'`, `LS_LAYOUTS_KEY`) — perdido ao limpar o browser ou trocar de máquina
2. **ProcessManager in-memory singleton** — perdido ao reiniciar o servidor Next.js/PM2

Resultado: toda vez que o servidor reinicia ou o browser recarrega, os terminais somem.

**Referência diagnóstico:** DEV 1 (2026-03-29) — análise de `page.tsx`, `TerminalPanel.tsx`, `usePtySocket.ts`, `process-manager.ts`

## Acceptance Criteria

- [ ] **AC1** — Nova tabela `command_room_terminals` criada no SQLite via migration em `src/lib/schema.ts`
- [ ] **AC2** — Ao fazer spawn de terminal (POST `/api/command-room/spawn`), registro é inserido na DB com status `active`
- [ ] **AC3** — Ao fechar terminal (POST `/api/command-room/kill`), status é atualizado para `closed` na DB
- [ ] **AC4** — Novo endpoint GET `/api/command-room/list` retorna terminais com `pty_status != 'closed'` da DB (não só do ProcessManager in-memory)
- [ ] **AC5** — Frontend abandona localStorage: no page load, busca terminais via API e restaura a lista
- [ ] **AC6** — Ao reconectar (reload), `usePtySocket` tenta conectar ao WS com o `id` existente da DB; se processo ainda estiver vivo no ProcessManager, reconecta; se não, spawna novo
- [ ] **AC7** — Layout dos terminais (cols, rows, posição) é persistido na DB e restaurado no reload
- [ ] **AC8** — Ao iniciar o servidor, ProcessManager verifica terminais com `pty_status = 'active'` na DB e atualiza para `crashed` os que não existem mais em memória

## Schema — Nova Tabela

```sql
CREATE TABLE IF NOT EXISTS command_room_terminals (
  id           TEXT PRIMARY KEY,           -- UUID v4 (mesmo ID do ProcessManager)
  agent_name   TEXT NOT NULL,              -- ex: '@dev', '@qa', 'shell'
  agent_display_name TEXT,                 -- ex: 'Dex', 'Quinn'
  project_path TEXT NOT NULL,              -- pasta do projeto
  cols         INTEGER NOT NULL DEFAULT 220,
  rows         INTEGER NOT NULL DEFAULT 50,
  pty_status   TEXT NOT NULL DEFAULT 'active'
               CHECK(pty_status IN ('active','idle','closed','crashed')),
  created_at   TEXT NOT NULL DEFAULT (datetime('now')),
  last_active  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_crt_status ON command_room_terminals(pty_status);
```

## Dev Notes

- Manter UUID como TEXT (mesmo ID do ProcessManager) — elimina mapeamento desnecessário
- `pty_status = 'crashed'` distingue "encerrado pelo servidor" de "fechado pelo usuário" — útil para UX (mostrar badge "reconectando..." em vez de sumir)
- O frontend deve mostrar terminais `crashed` com opção de "Reconectar" (spawna novo processo, mesmo agente/projeto)
- Não guardar scrollback na DB — é muito dado; ao reconectar, o xterm.js começa fresh ou usa o buffer in-memory se processo ainda vivo
