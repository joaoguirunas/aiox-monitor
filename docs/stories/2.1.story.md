# Story 2.1 — WebSocket Server

## Story

**Como** aiox-monitor,
**Quero** um servidor WebSocket rodando na mesma porta 8888,
**Para que** o frontend possa receber eventos em tempo real sem polling.

## Acceptance Criteria

- [ ] **AC1** — Arquivo `server.ts` na raiz do projeto inicia Next.js + WebSocket Server na porta 8888
- [ ] **AC2** — WebSocket aceita conexões em `ws://localhost:8888/ws`
- [ ] **AC3** — `npm run dev` executa `tsx server.ts` (não mais `next dev --port 8888`)
- [ ] **AC4** — `npm run build && npm start` também usa `server.ts` (produção)
- [ ] **AC5** — `src/server/ws-broadcaster.ts` exporta singleton com método `broadcast(message: WsMessage)`
- [ ] **AC6** — Broadcaster mantém Set de clientes conectados, remove automaticamente clientes desconectados
- [ ] **AC7** — Mensagens broadcast são JSON serializado com campo `type` obrigatório
- [ ] **AC8** — Servidor envia `{ type: 'ping' }` a cada 30s para manter conexões vivas
- [ ] **AC9** — `npm run typecheck` e `npm run build` passam sem erros

## Tasks / Subtasks

- [x] **Task 1** — Criar `server.ts` na raiz (AC: 1, 2, 3, 4)
  ```typescript
  import { createServer } from 'node:http';
  import { parse } from 'node:url';
  import next from 'next';
  import { WebSocketServer, WebSocket } from 'ws';
  import { setBroadcaster } from './src/server/ws-broadcaster';

  const dev = process.env.NODE_ENV !== 'production';
  const port = Number(process.env.PORT ?? 8888);

  const app = next({ dev, hostname: 'localhost', port });
  const handle = app.getRequestHandler();

  app.prepare().then(() => {
    const httpServer = createServer((req, res) => {
      const parsedUrl = parse(req.url!, true);
      handle(req, res, parsedUrl);
    });

    const wss = new WebSocketServer({ noServer: true });

    httpServer.on('upgrade', (req, socket, head) => {
      const { pathname } = parse(req.url ?? '');
      if (pathname === '/ws') {
        wss.handleUpgrade(req, socket as any, head, (ws) => {
          wss.emit('connection', ws, req);
        });
      } else {
        socket.destroy();
      }
    });

    setBroadcaster(wss);

    httpServer.listen(port, () => {
      console.log(`> Ready on http://localhost:${port}`);
    });
  });
  ```

- [x] **Task 2** — Criar `src/server/ws-broadcaster.ts` (AC: 5, 6, 7, 8)
  ```typescript
  import { WebSocketServer, WebSocket } from 'ws';

  export interface WsMessage {
    type: string;
    [key: string]: unknown;
  }

  let wss: WebSocketServer | null = null;

  export function setBroadcaster(server: WebSocketServer): void {
    wss = server;
    // Ping loop — keeps connections alive
    setInterval(() => {
      broadcast({ type: 'ping' });
    }, 30_000);
  }

  export function broadcast(message: WsMessage): void {
    if (!wss) return;
    const json = JSON.stringify(message);
    for (const client of wss.clients) {
      if (client.readyState === WebSocket.OPEN) {
        client.send(json);
      }
    }
  }

  export function getClientCount(): number {
    return wss?.clients.size ?? 0;
  }
  ```

## Dev Notes

### Por que Custom Server e não API Route?

Next.js App Router não suporta upgrade de protocolo HTTP→WebSocket em route handlers. A abordagem oficial para WebSocket em Next.js é um custom server que controla o `httpServer` diretamente.

### tsx --watch vs next dev

O `next dev` agora é chamado internamente pelo `server.ts` via `app.prepare()`. O `tsx --watch` observa mudanças em `server.ts` e reinicia o processo, mas o hot-reload do Next.js continua funcionando para as páginas e API routes.

### setBroadcaster — injeção de dependência

O `ws-broadcaster.ts` não importa `wss` diretamente — recebe via `setBroadcaster()`. Isso evita circular imports entre `server.ts` e os módulos server-side do Next.js.

### Ping/Pong

O `WebSocket` nativo do browser fecha conexões ociosas. O ping de 30s mantém a conexão viva mesmo sem eventos. O browser responde automaticamente ao ping do protocolo WS nativo do `ws` package.

### Compatibilidade de tipos

`server.ts` usa `import type` para Next.js onde possível. O cast `socket as any` no upgrade handler é necessário porque `node:http` e `ws` têm tipagens ligeiramente divergentes para o socket de upgrade.
