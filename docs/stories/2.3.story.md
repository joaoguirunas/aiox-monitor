# Story 2.3 — Broadcast de Eventos via WebSocket

## Story

**Como** frontend do aiox-monitor,
**Quero** receber automaticamente cada novo evento assim que o hook Python o registra,
**Para que** o Modo Lista e o Modo Kanban atualizem sem necessidade de polling ou refresh.

## Acceptance Criteria

- [ ] **AC1** — Ao processar um evento em `processEvent()`, o broadcaster emite `{ type: 'event:new', event, projectId, agentId? }`
- [ ] **AC2** — Ao atualizar o status de um agente em `trackAgent()`, o broadcaster emite `{ type: 'agent:update', agent, projectId }`
- [ ] **AC3** — Ao desativar um terminal, o broadcaster emite `{ type: 'terminal:update', terminal, projectId }`
- [ ] **AC4** — Broadcasts são fire-and-forget — falha ao broadcast não afeta o processamento do evento
- [ ] **AC5** — `useEvents` hook atualiza a lista local quando recebe `event:new` com `projectId` matching o filtro ativo
- [ ] **AC6** — `useAgents` hook atualiza o agente local quando recebe `agent:update`
- [ ] **AC7** — Modo Lista atualiza em tempo real: novo evento aparece no topo da tabela sem refresh
- [ ] **AC8** — O Modo Lista não faz polling — remove qualquer `setInterval` de refresh automático
- [ ] **AC9** — `npm run typecheck` passa sem erros

  ```typescript
  export function useEvents(filters: EventFilters = {}) {
    const [events, setEvents] = useState<Event[]>([]);
    const [loading, setLoading] = useState(true);
    const [total, setTotal] = useState(0);
    const { lastMessage } = useWebSocket();

    const filtersKey = JSON.stringify(filters);

    // Initial fetch
    useEffect(() => {
      setLoading(true);
      // ... fetch como antes ...
    }, [filtersKey]);

    // Real-time updates
    useEffect(() => {
      if (!lastMessage || lastMessage.type !== 'event:new') return;
      const msg = lastMessage as WsEventNew;
      // Only update if matches current project filter (or no filter)
      if (filters.projectId && msg.projectId !== filters.projectId) return;
      setEvents(prev => [msg.event, ...prev].slice(0, filters.limit ?? 100));
      setTotal(prev => prev + 1);
    }, [lastMessage]);

    return { events, loading, total };
  }
  ```

## Dev Notes

### Circular import — broadcaster em server modules

`ws-broadcaster.ts` está em `src/server/`. Os módulos `event-processor.ts` e `agent-tracker.ts` também estão em `src/server/`. Não há circular dependency porque `ws-broadcaster.ts` não importa nenhum deles.

### Broadcast no contexto Next.js

Os route handlers do Next.js rodam no processo Node.js do custom server. O `ws-broadcaster` é um singleton de módulo — a mesma instância é compartilhada entre o route handler e o WebSocket server. Isso funciona porque ambos estão no mesmo processo Node.js.

### Slice para evitar crescimento infinito

`[msg.event, ...prev].slice(0, limit ?? 100)` garante que a lista no browser não cresça indefinidamente com eventos em tempo real.

### Diferença projectId vs filters.projectId

`msg.projectId` é o ID numérico do projeto do evento recebido. `filters.projectId` é o filtro ativo na UI. Se o usuário não filtrou por projeto, ambos devem aparecer.
