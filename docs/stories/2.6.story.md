# Story 2.6 — Idle Detector

## Story

**Como** aiox-monitor,
**Quero** detectar automaticamente quando um agente fica sem atividade e transicionar seu status para `idle`, `break` e `offline`,
**Para que** o Modo Kanban e o Modo Lista reflitam o estado real dos agentes sem depender apenas de eventos explícitos.

## Acceptance Criteria

- [ ] **AC1** — `src/server/idle-detector.ts` exporta `startIdleDetector()` que inicia um loop de verificação a cada 30s
- [ ] **AC2** — Agente sem evento nos últimos `idle_timeout_lounge` segundos (default: 300s) → status muda para `idle`
- [ ] **AC3** — Agente `idle` sem evento por mais `idle_timeout_break` segundos (default: 900s) → status muda para `break`
- [ ] **AC4** — Agente `break` sem evento por mais `idle_timeout_offline` segundos (default: 3600s) → status muda para `offline`
- [ ] **AC5** — Transição de status faz broadcast de `{ type: 'agent:update', agent, projectId }` via `ws-broadcaster`
- [ ] **AC6** — `startIdleDetector()` é chamado em `server.ts` após `setBroadcaster(wss)`
- [ ] **AC7** — Ao receber qualquer evento de um agente, o status volta para `working` automaticamente (já implementado em `agent-tracker.ts` — verificar)
- [ ] **AC8** — `idle-detector` não cria registros duplicados — apenas atualiza agentes com status diferente do alvo
- [ ] **AC9** — `npm run typecheck` e `npm run build` passam sem erros

## Tasks / Subtasks

- [x] **Task 1** — Criar `src/server/idle-detector.ts` (AC: 1, 2, 3, 4, 5, 8)
  ```typescript
  import { broadcast } from './ws-broadcaster';
  import { getDb } from '@/lib/db';
  import type { Agent } from '@/lib/types';

  // Timeouts em segundos (lidos do banco ou defaults)
  const IDLE_TIMEOUT_LOUNGE = 300;   // working → idle (5 min)
  const IDLE_TIMEOUT_BREAK = 900;    // idle → break (15 min)
  const IDLE_TIMEOUT_OFFLINE = 3600; // break → offline (1h)

  const CHECK_INTERVAL_MS = 30_000; // checar a cada 30s

  export function startIdleDetector(): void {
    setInterval(checkIdleAgents, CHECK_INTERVAL_MS);
    console.log('[idle-detector] started');
  }

  function checkIdleAgents(): void {
    const db = getDb();

    // working → idle: sem evento há mais de IDLE_TIMEOUT_LOUNGE segundos
    const toIdle = db.prepare(`
      SELECT a.*, p.id as project_id
      FROM agents a
      JOIN projects p ON p.name = a.project_name
      WHERE a.status = 'working'
        AND (
          strftime('%s', 'now') - strftime('%s', a.last_active) > ?
        )
    `).all(IDLE_TIMEOUT_LOUNGE) as (Agent & { project_id: number })[];

    // idle → break: sem evento há mais de IDLE_TIMEOUT_BREAK segundos
    const toBreak = db.prepare(`
      SELECT a.*, p.id as project_id
      FROM agents a
      JOIN projects p ON p.name = a.project_name
      WHERE a.status = 'idle'
        AND (
          strftime('%s', 'now') - strftime('%s', a.last_active) > ?
        )
    `).all(IDLE_TIMEOUT_BREAK) as (Agent & { project_id: number })[];

    // break → offline: sem evento há mais de IDLE_TIMEOUT_OFFLINE segundos
    const toOffline = db.prepare(`
      SELECT a.*, p.id as project_id
      FROM agents a
      JOIN projects p ON p.name = a.project_name
      WHERE a.status = 'break'
        AND (
          strftime('%s', 'now') - strftime('%s', a.last_active) > ?
        )
    `).all(IDLE_TIMEOUT_OFFLINE) as (Agent & { project_id: number })[];

    updateAndBroadcast(toIdle, 'idle', db);
    updateAndBroadcast(toBreak, 'break', db);
    updateAndBroadcast(toOffline, 'offline', db);
  }

  function updateAndBroadcast(
    agents: (Agent & { project_id: number })[],
    newStatus: string,
    db: ReturnType<typeof getDb>
  ): void {
    if (agents.length === 0) return;

    const updateStmt = db.prepare(
      `UPDATE agents SET status = ? WHERE id = ?`
    );

    for (const agent of agents) {
      updateStmt.run(newStatus, agent.id);
      const updated = { ...agent, status: newStatus };
      try {
        broadcast({ type: 'agent:update', agent: updated, projectId: agent.project_id });
      } catch { /* fire-and-forget */ }
    }
  }
  ```

## Dev Notes

### Intervalo de 30s vs timeout real

O idle detector verifica a cada 30s mas os timeouts são de minutos. Isso é intencional — a precisão de ±30s é suficiente para este caso de uso. Não há problema em verificar frequentemente; a query é leve (agentes ativos são poucos).

### JOIN com projects

O `broadcast()` precisa de `projectId: number`. Agentes armazenam `project_name` (string), não `project_id`. Por isso o JOIN com `projects` na query.

### Agentes sem projeto correspondente

Se um agente não tem projeto correspondente (dados inconsistentes), o JOIN implícito excluirá esses agentes da verificação. Comportamento correto — não há como fazer broadcast sem `project_id`.

### Constantes vs company_config

Para MVP, usar constantes é suficiente. Se `company_config` já tem colunas de timeout (do schema existente), leia de lá. Caso contrário, não criar migration — as constantes funcionam bem.

### Agentes offline vs delete

Agentes offline não são deletados — apenas mudam de status. Isso permite histórico de atividade. Para limpar agentes muito antigos, criar task separada (fora do escopo desta story).

### Working status no receber evento

O `agent-tracker.ts` já deve definir `status = 'working'` ao receber qualquer evento (PreToolUse, PostToolUse, etc.). Verificar em Task 3 — se não estiver, adicionar. Isso garante que o ciclo working → idle → break → offline reinicia ao receber atividade.
