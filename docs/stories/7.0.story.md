# Story 7.0 — PM2 Fast Restart

## Story

**Como** operador do aiox-monitor,
**Quero** que o restart via PM2 seja o mais rápido possível e que o pipeline de build esteja correctamente separado do startup,
**Para que** o sistema permanente (PM2 com auto-start via launchd) reinicie sem overhead desnecessário após cada deploy de mudanças.

## Context

O aiox-monitor corre permanentemente via PM2 em background, auto-iniciando com o Mac via `pm2 startup launchd`. O ciclo de deploy é `npm run build && pm2 restart aiox-monitor`.

**Problema identificado:** O restart levava ~6-7s devido a 3 causas acumuladas:
1. `restart_delay: 3000` — 3s de espera fixa antes de iniciar
2. `script: 'npm', args: 'run start'` — PM2 → npm → shell → esbuild → node (4 camadas)
3. esbuild recompilava `server.ts` em cada restart, mesmo sem mudanças

**Solução aplicada:**
1. PM2 corre `node .server/server.mjs` directamente (zero intermediários)
2. esbuild movido para `npm run build` (compila uma vez, reutiliza)
3. `restart_delay` reduzido de 3000ms para 500ms

## Acceptance Criteria

- [x] AC1: `ecosystem.config.cjs` usa `script: '.server/server.mjs'` com `interpreter: 'node'`
- [x] AC2: `restart_delay` reduzido para 500ms (de 3000ms)
- [x] AC3: `npm run build` executa `esbuild server.ts → .server/server.mjs` seguido de `next build`
- [x] AC4: `npm run start` é simplesmente `node .server/server.mjs` (sem esbuild no startup)
- [x] AC5: `pm2 restart aiox-monitor` completa em < 1s (excluindo Next.js app.prepare())
- [x] AC6: `pm2 save` executado para persistir nova config no auto-start

## Technical Notes

### Antes (lento)
```
PM2 → npm run start → shell → esbuild server.ts → node .server/server.mjs → app.prepare()
      \___ 3s delay ___/  \______ ~1s overhead ______/  \______ ~0.2s _______/
```

### Depois (rápido)
```
PM2 → node .server/server.mjs → app.prepare()
 0.5s delay  \__ directo, sem intermediários __/
```

### Deploy workflow
```bash
npm run build && pm2 restart aiox-monitor
```
- `build`: esbuild server.ts (~10ms) + next build (~5-12s, varia com cache) = ~5-12s
- `restart`: PM2 reinicia node directo = ~0.3-0.4s (medido QA: 0.31-0.42s)
- **Total deploy: ~6-13s** (vs ~6-7s apenas para restart antes)
- **Ganho real:** restart isolado passou de ~6-7s para < 0.5s (-94%); build é one-time e paralelizável
