# Story 7.3 — Skins Server-Persisted

## Story

**Como** operador do aiox-monitor,
**Quero** que as skins dos agentes sejam guardadas no servidor e não no browser,
**Para que** não perca as configurações ao limpar dados do browser e as skins sejam consistentes em qualquer dispositivo que aceda ao dashboard.

## Context

As skins dos agentes são atribuídas na Config (`/empresa/config`) mas persistidas em `localStorage` via `SKIN_STORAGE_KEY = 'aiox-skin-config'` (`src/game/data/skin-config.ts:51-68`). A tabela `company_config` já é singleton (id=1), podemos adicionar uma coluna JSON para guardar os assignments.

O `getAgentSkin()` é chamado no `BootScene.ts` durante o load de assets — esta função precisa ser adaptada para ler do servidor em vez de localStorage.

### IDs de skins válidos (para validação API)

**Aliens (10):** `zyx`, `nebula`, `blip`, `blorp`, `nyx`, `pip`, `xara`, `ruk`, `flux`, `zara`
**Animals (10):** `cat`, `dog`, `panda`, `fox`, `lion`, `wolf`, `penguin`, `owl`, `horse`, `rabbit`

## Acceptance Criteria

### Schema & API

- [ ] AC1: **Schema migration** — Adicionar coluna `skin_assignments TEXT NOT NULL DEFAULT '{}'` à tabela `company_config` em `src/lib/schema.ts`. Usar padrão de migration defensiva existente: `try { db.exec('ALTER TABLE ...'); } catch { /* already exists */ }`
- [ ] AC2: **Type update** — Adicionar `skin_assignments: string` (JSON text) ao interface `CompanyConfig` em `src/lib/types.ts:77-89`
- [ ] AC3: **API PUT validation** — Em `src/app/api/company-config/route.ts`, aceitar campo `skin_assignments` no body. Validar: deve ser JSON string válido, cada key deve começar com `@`, cada value deve ser um skinId válido (existir em `ALL_SKINS`) ou `'default'`. Rejeitar com 400 se inválido
- [ ] AC4: **API GET** — `GET /api/company-config` já retorna todos os campos — confirmar que `skin_assignments` vem no response. `updateCompanyConfig` em `queries.ts` deve incluir `skin_assignments` nos campos actualizados

### Frontend

- [ ] AC5: **Config page load** — Substituir `loadSkinConfig()` (localStorage) por leitura do campo `skin_assignments` do response de `GET /api/company-config`. Parse JSON. Fallback para `{}` se parsing falha
- [ ] AC6: **Config page save** — No `handleSave()`, enviar `skin_assignments: JSON.stringify(skinConfig)` no body do PUT. Remover chamada a `saveSkinConfig()` (localStorage)
- [ ] AC7: **Migração one-time** — Na Config page, após carregar config do servidor: se `skin_assignments` no servidor está vazio (`'{}'`) E localStorage tem dados (via `loadSkinConfig()`), copiar os dados do localStorage para o state local. No próximo save, vão para o servidor. Mostrar toast informativo: "Skins migradas do browser para o servidor"

### Game Engine

- [ ] AC8: **BootScene adaptation** — `getAgentSkin()` em `src/game/data/skin-config.ts` actualmente lê de localStorage. Modificar para aceitar um `SkinAssignment` como parâmetro (injectado pelo PhaserGame wrapper), com fallback para localStorage (backwards compatibility durante transição). O `PhaserGame.tsx` já faz fetch de `/api/company-config` — passar `skin_assignments` parsed para o Phaser via bridge

### Cleanup

- [ ] AC9: **Deprecate localStorage** — Após a migração one-time (AC7) ter corrido com sucesso, remover `SKIN_STORAGE_KEY` do localStorage (`localStorage.removeItem(SKIN_STORAGE_KEY)`). Manter as funções `loadSkinConfig` / `saveSkinConfig` com `@deprecated` annotation até que todos os consumers estejam migrados

### JSON format do skin_assignments

```json
{
  "@dev": "cat",
  "@qa": "zyx",
  "@architect": "nebula"
}
```

Agentes sem skin custom simplesmente não aparecem no objeto (ou têm valor `"default"`).

### Validação server-side (pseudocódigo)

```typescript
if (body.skin_assignments !== undefined) {
  try {
    const parsed = JSON.parse(body.skin_assignments);
    if (typeof parsed !== 'object' || Array.isArray(parsed)) throw new Error();
    for (const [key, val] of Object.entries(parsed)) {
      if (!key.startsWith('@')) throw new Error(`Invalid agent: ${key}`);
      if (val !== 'default' && !ALL_SKINS.some(s => s.id === val)) throw new Error(`Invalid skin: ${val}`);
    }
  } catch {
    return NextResponse.json({ error: 'Invalid skin_assignments' }, { status: 400 });
  }
}
```
