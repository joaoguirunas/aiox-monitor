# Story 8.6 — Frontend Terminal Removal via WebSocket

## Story

**Como** utilizador do aiox-monitor,
**Quero** que terminais removidos ou desactivados desapareçam da UI em tempo real,
**Para que** a aba Terminais não acumule cards fantasma quando mantenho o tab aberto.

## Context

**Bug actual:** O WebSocket handler em `useTerminals.ts` adiciona novos terminais ao state React (`unshift`) e actualiza existentes, mas nunca remove terminais que foram eliminados pela `cleanupStaleTerminals()` ou desactivados. A lista só é limpa num full refetch (ao mudar de projecto ou clicar "Atualizar").

**Referência:** `docs/Terminais.md` secção 5.6 | Epic 8 Bug Ref 5.6

## Acceptance Criteria

### Task 1: Novo tipo de mensagem WebSocket
- [ ] **AC1** — Adicionar interface `WsTerminalRemoved` em `src/lib/types.ts`:
  ```typescript
  export interface WsTerminalRemoved extends WsMessage {
    type: 'terminal:removed';
    terminalId: number;
    projectId: number;
  }
  ```
- [ ] **AC2** — Adicionar `WsTerminalRemoved` ao union type `WsIncomingMessage`

### Task 2: Broadcast de remoção no backend
- [ ] **AC3** — Em `cleanupStaleTerminals()` (`src/server/terminal-tracker.ts`), antes de chamar `deactivateStaleTerminals()` e `purgeOldInactiveTerminals()`, obter a lista de terminais que vão ser afectados
- [ ] **AC4** — Após a deactivação/purge, para cada terminal que mudou para `inactive` ou foi eliminado, fazer broadcast de `{ type: 'terminal:removed', terminalId, projectId }`
- [ ] **AC5** — Em `deactivateTerminal()` (chamada por syncSystemTerminals), também broadcast `terminal:removed` quando o terminal passa a `inactive`

### Task 3: Handler no frontend
- [ ] **AC6** — Em `useTerminals.ts`, adicionar handler para `terminal:removed`: filtrar o terminal do state React
  ```typescript
  if (lastMessage.type === 'terminal:removed') {
    setTerminals(prev => prev.filter(t => t.id !== msg.terminalId));
  }
  ```
- [ ] **AC7** — O handler respeita o filtro de `projectId` (se activo, só remove terminais do projecto seleccionado)

### Task 4: Validação
- [ ] **AC8** — Abrir aba Terminais, manter tab aberto. Após 15min de inactividade dum terminal, o card desaparece automaticamente (sem refresh manual)
- [ ] **AC9** — Ao fechar Claude Code, após cleanupStaleTerminals correr, o card transita para inactive e eventualmente desaparece
- [ ] **AC10** — `npm run build` passa sem erros

## Dev Notes

### Impacto no `terminal:update` existente
Quando um terminal passa a `inactive` via `terminal:update`, o frontend já o mostra na secção "Inativos". O `terminal:removed` é complementar — remove terminais que são purgados (eliminados da DB) ou que o utilizador não precisa de ver.

**Decisão:** Emitir `terminal:removed` apenas quando o terminal é efectivamente ELIMINADO da DB (por `purgeOldInactiveTerminals`), ou quando passa a `inactive` por `deactivateStaleTerminals`. Para transições normais de status, `terminal:update` já é suficiente.

### Alternativa considerada
Periodic refetch (setInterval no frontend) — rejeitado porque duplicaria tráfego de rede e não é consistente com a arquitectura WS-first do projecto.
