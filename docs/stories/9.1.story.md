# Story 9.1 — API de Sessões + Paginação Server-Side

## Story

**Como** utilizador do aiox-monitor,
**Quero** que o backend exponha as sessões com dados agregados (prompt, resposta, tools) via endpoint dedicado,
**Para que** o frontend possa mostrar o histórico completo de sessões sem depender de agrupamento client-side sobre dados truncados.

## Context

Actualmente a Lista busca eventos individuais com `LIMIT 200` (`GET /api/events`) e agrupa-os client-side via `groupBySession()`. Com sessões de 100+ eventos, o limit esgota-se nas sessões mais recentes — apenas 3 de 23 sessões são visíveis (87% do histórico invisível). A sessão 739 com 162 eventos chega truncada com apenas 38 eventos.

A tabela `sessions` já existe no banco com dados ricos (`event_count`, `status`, `started_at`, `ended_at`) mas não tem endpoint API. Esta story cria o endpoint que permite ao frontend paginar por sessões (não por eventos) e carregar eventos de uma sessão on-demand.

## Dev Notes

### SQL de referência para getSessions

```sql
SELECT s.*,
  (SELECT e.input_summary FROM events e
   WHERE e.session_id = s.id AND e.type = 'UserPromptSubmit'
   ORDER BY e.id ASC LIMIT 1) as prompt,
  (SELECT e.input_summary FROM events e
   WHERE e.session_id = s.id AND e.type IN ('Stop', 'SubagentStop')
   ORDER BY e.id DESC LIMIT 1) as response,
  (SELECT COUNT(*) FROM events e
   WHERE e.session_id = s.id AND e.type = 'PreToolUse') as tool_count,
  (SELECT GROUP_CONCAT(DISTINCT e.tool) FROM events e
   WHERE e.session_id = s.id AND e.tool IS NOT NULL
   AND e.type IN ('PreToolUse', 'PostToolUse')) as tools
FROM sessions s
WHERE {conditions}
ORDER BY s.started_at DESC
LIMIT ? OFFSET ?
```
