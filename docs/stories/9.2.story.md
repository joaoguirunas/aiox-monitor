# Story 9.2 — Vista de Sessões com Dados Completos

## Story

**Como** utilizador do aiox-monitor,
**Quero** que a vista "Sessões" na Lista mostre todas as sessões do banco com paginação real,
**Para que** nenhuma sessão seja invisível ou truncada e eu tenha acesso ao histórico completo de comandos dos agentes.

## Context

Com a Story 9.1, o backend expõe sessões com dados agregados. Agora o frontend precisa de consumir esse endpoint em vez de fazer agrupamento client-side sobre eventos truncados.

Actualmente `page.tsx` (L20) usa `useEvents({ limit: 200 })` + `groupBySession(events)` (em `SessionRow.tsx` L20-68) — o que resulta em apenas 3 de 23 sessões visíveis. Esta story substitui esse flow por `useSessions()` que consome `/api/sessions` directamente.

**Componentes a modificar:**
- `src/app/lista/page.tsx` — usar `useSessions()` no modo summary
- `src/components/lista/SessionTable.tsx` — aceitar `SessionWithSummary[]`
- `src/components/lista/SessionRow.tsx` — renderizar dados do endpoint
- `src/components/lista/SessionDetail.tsx` — carregar eventos on-demand

## Dev Notes

### Padrão de referência

`useSessions` deve seguir exactamente o padrão de `useEvents.ts` (L1-71):
- `buildQueryString()` para serializar filtros
- `useEffect` para fetch inicial com cancellation token
- `useEffect` para WebSocket updates
- Estado: `sessions`, `loading`, `total`, `refreshKey`

### Componentes existentes que mudam de input

| Componente | Antes (SessionGroup) | Depois (SessionWithSummary) |
|-----------|---------------------|----------------------------|
| `SessionTable` | `sessions: SessionGroup[]` | `sessions: SessionWithSummary[]` |
| `SessionRow` | `session.prompt` (do groupBySession) | `session.prompt` (do endpoint) |
| `SessionDetail` | `session.events` (array em memória) | Fetch on-demand via API |

A interface `SessionWithSummary` já tem os mesmos campos que `SessionGroup` usa — a transição é natural.

### WebSocket strategy

Para manter real-time sem refetch constante:
- `event:new` com `session_id` existente → patch local (increment event_count)
- `event:new` com `session_id` novo → refetch página 1 (nova sessão aparece no topo)
- Reconexão WS → full refetch
