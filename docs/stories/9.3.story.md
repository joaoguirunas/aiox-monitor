# Story 9.3 — Paginação de Eventos Individuais

## Story

**Como** utilizador do aiox-monitor,
**Quero** navegar todos os eventos individuais do banco com paginação real,
**Para que** as vistas "Todos", "Prompts", "Respostas", "Tool Calls" e "Resultados" mostrem o histórico completo sem truncamento.

## Context

As vistas de eventos individuais (`all`, `UserPromptSubmit`, `Stop`, `PreToolUse`, `PostToolUse`) usam `useEvents({ limit: 200 })` — um hardcode em `page.tsx:20` que limita a 200 eventos. Com 522 eventos no banco, mais de metade é invisível. Não existe mecanismo de "carregar mais".

Esta story adiciona paginação incremental ao `useEvents` hook e remove o limit hardcoded, permitindo que o utilizador navegue todo o histórico evento por evento.

## Dev Notes

### Mudanças no useEvents

O hook actual (L19-71) já tem `total` no retorno mas não `hasMore` nem `loadMore`. Adicionar:

```typescript
const [loadingMore, setLoadingMore] = useState(false);

const loadMore = useCallback(() => {
  if (loadingMore || !hasMore) return;
  setLoadingMore(true);
  const nextOffset = events.length;
  fetch(`/api/events?${buildQueryString({ ...filters, offset: nextOffset })}`)
    .then(r => r.json())
    .then(data => {
      setEvents(prev => [...prev, ...data.events]);
      setLoadingMore(false);
    });
}, [events.length, hasMore, loadingMore, filtersKey]);
```

### WebSocket + paginação

Novos eventos via WS são prepended (mais recentes no topo). Load more traz eventos mais antigos (append no fundo). Não há conflito — os ranges não se sobrepõem.

### Impacto em page.tsx

A vista `summary` agora usa `useSessions` (Story 9.2). O `useEvents` é usado apenas nas vistas `all`, `UserPromptSubmit`, `Stop`, `PreToolUse`, `PostToolUse`.
