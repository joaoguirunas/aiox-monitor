# Story 9.4 — Contadores Fiéis e Indicadores de Completude

## Story

**Como** utilizador do aiox-monitor,
**Quero** ver contadores precisos que reflectem os totais reais do banco e indicadores claros de paginação,
**Para que** eu saiba exactamente quanta informação existe e quanta estou a ver.

## Context

Actualmente o header da Lista mostra "3 sessões de 522 eventos (agrupado)" — mistura unidades (sessões vs. eventos) e não reflecte o total real de sessões (23). Com as Stories 9.2 e 9.3, a paginação é real mas os contadores e indicadores precisam de ser unificados e corrigidos.

## Acceptance Criteria

### Task 1: Contadores por vista

- [x] **AC1** — Vista summary: "X sessões de Y total" (Y = total de sessões no banco para os filtros activos)
- [x] **AC2** — Vista all/tipo: "X eventos carregados de Y total" (Y = total real do endpoint)
- [x] **AC3** — Quando X = Y, não mostrar "de Y" (ex: "23 sessões" em vez de "23 sessões de 23 total")

### Task 2: Indicador de paginação

- [x] **AC4** — Quando `hasMore === true`, mostrar texto subtil: "Mostrando X de Y — carregar mais" como link clicável
- [x] **AC5** — Link "carregar mais" no indicador chama a mesma `loadMore()` do botão

### Task 3: Indicador de retenção

- [x] **AC6** — Badge discreto no header: "Retenção: 30 dias" (valor lido de `company_config.event_retention_days` via novo endpoint ou endpoint existente)
- [x] **AC7** — Se `event_retention_days` não estiver disponível, não mostrar o badge (graceful)

## Dev Notes

### Endpoint para config

Verificar se já existe endpoint que retorna `company_config`. Se não, a retenção pode ser um valor hardcoded no frontend (30 dias) com TODO para ler do backend — manter scope pequeno.

### Unidades correctas

O ponto-chave é nunca misturar "sessões" com "eventos" no mesmo contador. Cada vista tem a sua unidade:
- Summary → sessões
- All/tipo → eventos
